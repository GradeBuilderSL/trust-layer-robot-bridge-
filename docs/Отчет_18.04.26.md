## 4. Почему robot-bridge не установился сразу

Причины были организационные и инфраструктурные одновременно:

- на Jetson были перепутаны каталоги:
  - ожидался `/opt/trust-layer-bridge`
  - рабочая свежая копия оказалась в `/home/noetix/trust-layer-robot-bridge-`
  - при этом `/opt/trust-layer-bridge` существовал, но был почти пустой
- из-за этого `scp` сначала падал: не существовали нужные подкаталоги
- часть документации и скриптов ссылалась на старый путь `/opt/...`
- `bridge.main` сначала не запускался из-за отсутствия `fastapi`
- shell-скрипты после копирования с Windows содержали `CRLF`
- helper сначала не собирался из-за отсутствующей линковки generated DDS symbols
- guest Wi-Fi блокировал peer-to-peer доступ между ноутбуком и роботом
- DNS/apt на Jetson сначала были неисправны, что мешало ставить пакеты

## 5. Почему Trust Layer пока не работает как исполнительный контур

Важно разделять:

- интеграция работает
- робот не исполняет движения

Что уже подтверждено:

- `e1_server` отвечает с `transport=noetix_dds`
- helper стартует
- HTTP-команды доходят до helper
- helper публикует DDS-команды

Что не подтверждено:

- реальное чтение `Robot_Status_Topic`
- реальный текущий `workmode`
- реальный переход робота в walking/preparation/user mode
- реальный исполнительный отклик на `Control_Cmd_Topic`

То есть на данный момент проблема не в HTTP-слое Trust Layer, а в неподтвержденном нижнем исполнительном слое робота.

## 7. Зачем был написан C++ helper

SDK E1 — это не готовый HTTP API, а C++ DDS SDK с generated IDL типами и vendor-библиотеками.

Почему Python-решение было недостаточным:

- Python не имел готового надёжного доступа к vendor-типам
- требовались symbols из `libdriver.a`, `libddsc.so`, `libddscxx.so`
- безопаснее было использовать уже существующие vendor serializer'ы и DDSWrapper

По сути, C++ helper — это тонкий мост между Python HTTP слоем и нативным DDS SDK робота.

## 8. Есть ли стабилизирующий/подготовительный модуль, который может блокировать ходьбу

Прямого слова `stabilizer` в SDK нет, но признаки такого слоя есть.

### Что найдено в SDK

- в `lowcontroller.cpp` загружается `policy_walk.onnx`
- в SDK есть:
  - `policy_walk.onnx`
  - `policy_walk.rknn`
  - `policy_user.onnx`
  - `policy_user.rknn`
- в `highcontroller.h` есть `WorkMode`:
  - `STAND`
  - `LIE`
  - `USERWALK`
  - `DEFAULT`
- в `lowcontroller.cpp` реализована state machine:
  - `startcontrol`
  - `LIE`
  - `STAND`
  - `USERMODE`
  - `TO USERWALK MODE`
  - `WALK2STAND`

### Вывод

Высока вероятность, что перед реальной локомоцией робот должен пройти подготовительный контур:

- включение control
- переход в `LIE`
- затем в `STAND`
- затем в `USERWALK` / locomotion policy

Если этот контур не активирован, высокоуровневые команды движения могут игнорироваться.

## 9. Почему прямые команды мимо Trust Layer тоже не сработали

Это один из самых важных симптомов.

Мы запускали vendor `highcontrol`, и робот тоже не реагировал. Значит:

- проблема не только в Trust Layer
- проблема либо в текущем режиме робота
- либо в неактивированном policy/state machine слое
- либо в том, что `highcontrol` сам не поднимает робота в locomotion-ready состояние

Из кода видно:

- `highcontroller.cpp` публикует `cmode.mode(1)`
- затем публикует `ControlCmd` по данным джойстика
- но полноценную последовательность `lie -> stand -> userwalk` реализует не он, а `lowcontroller.cpp`

Это делает гипотезу про обязательный подготовительный контур очень сильной.

## 12. Правильно ли мы действуем

В целом да, но с поправкой.

Что было сделано правильно:

- мы пошли в реальный DDS слой робота
- не стали выдумывать несуществующий vendor API
- не стали сразу переписывать локомоцию с нуля
- доказали, что команды доходят до DDS

Где была завышенная гипотеза:

- мы предположили, что `Control_Cmd_Topic` сам по себе уже достаточен для движения
- SDK показывает, что ниже есть ещё режимный и policy слой

Вывод:

локомоцию с нуля писать пока не нужно. Скорее всего, нужно правильно встроиться в уже существующий low-level policy / mode pipeline.
